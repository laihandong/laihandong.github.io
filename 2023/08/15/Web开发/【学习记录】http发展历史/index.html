历史

传输层

这里拿比较古老但最具代表性的OSI（开放式系统互联）作为参考，来缕清数字信息在终端之间相互传输的过程中，哪里用到了http

http协议出现于应用层，必须经过传输层才能进行通信。理论上，使用传输层的哪个协议对http来说并不重要，但是传输层本身对http至关重要。

TCP/UDP

定义

transmission control protocol传输控制协议，简称tcp，使用tcp协议的http版本有http0.9/1.0/1.1/2.0

user datagram protocl用户数据报协议，简称udp，使用udp协议的http版本有http3.0

http0.9

http0.9作为最简陋的协议，仅使用单行请求信息，仅支持响应html资源，没有标头、状态码、错误码等等

http1.0

http1.0：请求中增加了http协议标头，响应中增加了状态码和传输内容类型（即服务器配置的MIME类型–用于服务器解析文件类型用以支持更多类型的数据传输）。但此时一个连接只能处理一个请求一个响应，避免不了频繁的建立物理连接带来的成本问题（因为建立单次物理连接的成本很高）。

http1.1

持久连接

引入了持久连接的机制，一个连接中可以处理多次请求和响应，节省了（tcp）大部分挥手/握手时间。

流水线

同时它也引入了流水线的机制，即一次多个请求逐个响应（在服务器响应前发送多个请求并等待服务器顺序响应），节省了部分请求发送的时间。

线头阻塞

流水线机制存在一个重要缺陷–head of line blocking线（队）头阻塞。比如顺序发送了3个请求，第一个请求处理时间过长，那么后面的请求就必须等待前者处理完才能响应。常见的处理方法就是准备多个连接备用，当有连接阻塞，可以启用备用连接发送请求。

标头冗余

同时另一个问题就是**http标头冗余问题**，大多每次请求都会带同样的标头，该问题在这个版本并没有得到解决

http2

该版本更好地解决了http1.1遗留的问题和进行了功能拓展，而不是作为http1.1的替代。

改变传输方式

http2在应用层改变了传输方式–之前http协议在应用程序层传递的标头和数据都是明文的，http2在应用程序层增加了一个二进制帧数据处理层，将标头和数据拆分成二进制数据帧进行传输。每个数据帧包含了自己携带的数据并用stream id做标识，数据被拆分为多个数据帧形成数据流，经打包后传输到目的地，再根据stream id组装和还原原始数据。

好处

好处就是二进制数据很适合应用程序层下的底层协议，原本属于它们的大部分工作被应用程序层所承担了，减少了底层的工作量，提高了网络传输的速度。无论多大的文件都能被拆分为无数个数据帧，这样同时也规避了大文件传输大小的限制。

多路复用

此时，http2协议是一个客户端与服务器建立一个连接，之间通过双向数据流进行通信，并且可以重用单个连接传递数据，以此形成http2的多路复用–一个客户端和一个服务器保持一个链接，在该连接上并发传输，就是多路复用。

线头阻塞问题

由于数据完整性可以根据stream id进行校验，所以数据帧不一定是顺序传输，先到的数据帧会先进入缓存并进行组装和还原，这样也就解决了http1.1应用程序层的http线头阻塞问题。

但是http2并没有解决传输层的线头阻塞问题，原因如下：

http2除了在应用程序层中将数据“剁碎”并加入stream id用数据流的方式传递，其它的http2并不关心，更不用说数据流到达更为底层的传输层是如何传递的，同时传输层中的tcp协议也并不关心该数据流传输的是不是http响应信息。tcp只知道使用最大大小的数据包（一般是1450字节）来跟踪被赋予它传输的指定字节范围的数据部分，以便可以按正确顺序还原原始数据。

所以当使用tcp协议的传输层中，有数据包丢失时，至少得重复请求一次服务器并得到丢失数据的副本后，才能继续解包其它tcp数据再交给应用程序层，这种情况则会导致（传输层）tcp的线头阻塞，结论就是http2未能完全解决线头阻塞问题

服务器推送

服务器可以将未经过请求的的页面所需js css等文件资源一同推送给客户端，这样则进一步减少了网络请求的数量。同时也更好地支持了服务器发送事件server send events机制，提高了前端开发效率。

压缩标头

http2使用HPACK来编解码http标头，可以起到压缩标头的效果，进一步提高网络传输效率，增强交互。

HPACK原理：客户端/服务端双方都维护着相同的61个条目的只读静态表和可动态添加的空白的动态表。

客户端请求发送时先查找表中的键值对，直接提取该标头在表中对应的的整数索引，表示该键值对。如果只匹配名称，则使用索引表示名称，值则用霍夫曼算法进行编码后表示。表中没有的则按照顺序添加到动态表中，再使用霍夫曼算法编码键值对字符后传递给服务器。

服务端响应时则相反，使用表中的索引解码，或者使用霍夫曼算法进行解码后添加到服务器动态表中，这样保证双方的表中保存的数据和整数索引是一致的。在之后请求的标头会越来越小，

思考

  • 是否能基于hpack原理更好地进行优化网络请求？

    ​ 交互最好是同域名或者同域名下不同二级域名指向的服务器ip是一样的，而且使用的是同一个加密证书，使用的是同一静动态标头表。

  • hpack按顺序编解码标头，会不会造成线头阻塞？

    是的。因为请求标头数据流可以是无序的，服务端解码动态标头表无法解析最大索引值之外的请求标头索引，直到携带该索引对应的正确霍夫曼值的数据流打到服务器。该问题目前无法解决，只能留给http3了

  • 使用多路复用真的比之前版本中的顺序传输更快吗？并真的不会有其它线头阻塞问题吗？

    不一定。因为http2和现代浏览器被强制必须使用https加密超文本协议传输，多出了至少2rtt的tls时间。其次，顺序传输有的情况会比乱序快，需要视具体情况使用优先控制来优化。其次，如果数据包丢包了，刚好有一个数据帧丢失，虽然没有阻塞其它消息，但阻塞了自己，这中基于数据流的线头阻塞不仅存在于http2中，也会发生在http3中，这是一个概率问题。

http3

它是基于quic协议的升级版本，即quic + http2 = htt3。

由于http2最多只能解决应用程序层的线头阻塞，而传输层的协议基本由系统实现，应用程序层无法干涉。所以http3直接**抛弃了tcp，在传输层使用udp，同时优化了一个协议用于在应用程序层适配udp并保证数据传输的可靠性，就推出了基于udp的quic协议，也被称为应用层的tcp2.0**，同时也保留了http2基于流的多路复用和http2的其它优化功能。

基于udp的流

http3将基于udp的流视为“一等公民”，与http2中的流有两个重要区别。

一是动态标头编码，http3使用qpack编解码。二是qpack解决了hpack按顺序解码容易造成线头阻塞问题–使用指令独立流使客户端/服务端双方确定标头是否完成同步，以解决动态标头的塞车问题。

规避类似tcp线头阻塞问题

http3优化了数据帧的组成部分，脱掉了tcp“外衣”，直接在数据帧中加入了数据的offset字节范围帧头，即使丢了某个数据包，也不影响其它完整信息的还原，消除了tcp线头阻塞的缺点。

优点

http3在传输层使用udp进行数据传输，相比tcp进一步优化了交互数据大小，提高了传输速度

udp在建立连接上，不仅少了三次握手、四次挥手的时间，也减少了加密协商的时间，提高连接速度。

quic协议使用随机数作为connection id（即连接id），来识别客户端和同域名服务器建立的连接，从而使连接不受网络变化的影响–这是相对于tcp基于四元组（源ip、源端口、目标ip、目标端口）来说的。同时也保留了双向流多路复用的功能。
